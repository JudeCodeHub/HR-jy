# MasterHR: The Ultimate "A to Z" Technical & Operational Guide

This document explains the "How," "Why," and "What" of the MasterHR project, covering every detail from the first line of code to the final user interaction.

---

## 1. The Core Vision (The "Why")
Traditional HR systems rely on passwords or physical ID cards for attendance. Both are flawed: passwords can be shared, and cards can be "swiped" by a colleague (Buddy Punching). MasterHR solves this by using **Biological Identity**. By combining Face Recognition with **Liveness Detection**, we create a system that is impossible to cheat without the physical presence of the employee.

---

## 2. Phase A: Identity Initialization (Data Seeding)
Before the system can recognize anyone, it needs a "Memory."
1. **Employee Registration**: HR adds an employee via the mobile app. A high-quality "Anchor Image" is captured.
2. **Encoding Generation**: The `encodings.py` script processes these images.
    - It uses **dlib’s HOG (Histogram of Oriented Gradients)** to find the face.
    - It maps **128 key points** (landmarks) on the face (eye distance, nose bridge length, jawline, etc.).
    - These 128 numbers are the "Digital Signature" of that person.
3. **Serialization**: These signatures are saved into `EncodeFile.p` (a Pickle file). This allows the AI to load the "Memory" in milliseconds rather than re-processing every photo every time.

---

## 3. Phase B: Secure Access (Authentication)
1. **Login**: Uses a standard Email/Password flow.
2. **JWT Security**: The Node.js backend generates a **JSON Web Token**.
    - This token is "Signed" with a secret key.
    - The mobile app stores this token locally.
    - **Why?**: This prevents "Man-in-the-Middle" attacks. Even if someone intercepts a request, they cannot fake the user's identity without the secret key.

---

## 4. Phase C: The Attendance Workflow (The "A to Z" Flow)
This is the most complex part of the system.

### Step 1: Geofencing Verification
- The app fetches the office's latitude/longitude from the Node.js server.
- It calculates the distance between the user's current GPS and the office.
- If the user is > 100 meters away, the "Mark Attendance" button is disabled.

### Step 2: Liveness Detection (Antispoofing)
- **The Problem**: A user might show a photo of a colleague to the camera.
- **The Solution**: The Flask server uses a **Convolutional Neural Network (CNN)**.
    - The model (`1.h5`) was trained on thousands of "Real" vs "Fake" faces.
    - It looks for "spoofing cues" like screen reflections, paper edges, or lack of skin texture depth.
    - If it detects a photo/screen, it returns "You are not Live."

### Step 3: Face Matching
- If live, the `face_recognition` library compares the 128-point signature of the *current* camera feed with the signatures in `EncodeFile.p`.
- It calculates the **Euclidean Distance** (mathematical difference). A distance < 0.4 usually means a perfect match.

### Step 4: Final Validation
- The system checks if the detected face matches the **Logged-in User ID**. You cannot mark attendance for someone else even if their face is recognized.

### Step 5: Database Commit
- Once all checks pass, the Node.js server creates an entry in the `Attendance_record` collection with a timestamp and "Present" status.

## 5. Phase D: Operational Modules (Management)

### Task Management Logic
- **HR Side**: Can create tasks with a "Deadline."
- **Employee Side**: Tasks are sorted by priority/date.
- **Why?**: This bridges the gap between "being at work" (attendance) and "doing work" (productivity).

### Payroll System
- Links attendance records with a salary multiplier.
- Automatically flags "Late" or "Absent" days to adjust the final payout.
- Statuses: `Pending`, `Paid`, `Review Required`.

### Real-Time Communication
- A lightweight chat system.
- **Schema**: Stores `senderId`, `receiverId`, `message`, and `timestamp`.
- Allows instant clarification on tasks or leave requests.

## 6. Technical Stack Rationale (The "How It's Built")

| Technology | Role | Why This? |
| :--- | :--- | :--- |
| **React Native** | Frontend | Cross-platform (iOS/Android) with one codebase. Very fast UI rendering. |
| **Node.js** | Backend API | Non-blocking I/O. Perfect for handling hundreds of concurrent employee requests. |
| **Flask (Python)** | AI Engine | Python has superior support for ML (TensorFlow) and CV (OpenCV). |
| **MongoDB** | Database | Flexible schemas. Easy to add new employee fields without breaking the DB. |
| **Pickle** | Data Storage | Extremely fast way to store/load complex Python objects (encodings). |

---

## 7. Security & Privacy Measures
- **Data Encryption**: Passwords are never stored in plain text (Bcrypt).
- **Session Expiry**: JWT tokens expire, forcing a re-login for security.
- **Local Encodings**: Only the numerical encodings are used for comparison, not the raw images (where possible), protecting privacy.

---

## 8. Summary of the "A to Z" Lifecycle
1. **Admin** registers **Employee**.
2. **System** learns the **Employee's Face**.
3. **Employee** enters the **Office Zone** (GPS).
4. **Employee** verifies **Liveness** and **Identity** (AI).
5. **System** records **Attendance**.
6. **Admin** monitors **Tasks** and **Performance**.
7. **System** calculates **Payroll**.
8. **Cycle repeats daily.**
